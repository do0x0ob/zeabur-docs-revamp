# Zeabur Docs Revamp

## 核心觀點：文件有兩種讀者

Zeabur 作為一個 AI 驅動的產品，我相信文件不只是給人看的，也是給 AI 看的。但這兩個讀者需要不同的東西。

以我自己為例——接觸新平台時，我通常直接試用，卡住了才看 Quickstart，而且注意力很短。我期望用最少時間搞懂「這東西怎麼用」。所以 Quickstart 必須在最短篇幅內，讓我從零到「哦，這好用」。更深入的東西，等我確認這個體驗存在之後再說。

所以我認為 Quickstart 永遠是最重要的文件。隨著 AI Agent 的發展，用戶會越來越依賴 AI 直接從 Docs 找答案，而不是自己讀文檔。因此我認為需要將文件分為兩種類型：

### Human-first 文件

適用場景：需要在 UI 裡操作的流程（建立帳號、點按鈕、設定域名）

因為 AI agent 操作瀏覽器還不夠穩定，這類文件的讀者主要還是人，需要：

- 截圖與視覺引導
- CTA 按鈕連結到 App 內對應位置
- Step-by-step，不跳步驟
- 先讓用戶體驗成功，再解釋原理

### AI-first 文件

適用場景：部署設定、框架適配、API 參考、錯誤排除

這類文件的主要消費者是 Zeabur 的 AI 對話窗口和外部 AI agent。需要：

- 一問一標題（方便語意索引）
- 結構化表格和清單
- 關鍵字精準、術語一致
- 直接附 code example

## 三層內容生態

| 層 | 用途 | 特性 |
|----|------|------|
| Docs | 工具書 | 結構化、可索引、AI 可讀 |
| Blog | 故事與案例 | 深度內容、SEO、品牌 |
| Forum | 長尾排障 | 社群驅動、減少重複支援 |

這次重構聚焦在 Docs 層，目標是跟 Blog 和 Forum 互補而不是搶功能。

---

## 發現的問題

現有技術文件的內容品質不錯，各框架和功能的覆蓋都齊全。主要的問題在架構：好內容被埋在不直覺的分類裡，用戶和 AI 都不容易找到。

1. 沒有真正的 Quickstart：現有 get-started 花太多篇幅在註冊流程，帳號註冊對多數用戶來說已經很直覺，篇幅應該放在讓用戶最快體驗到產品的「魔法時刻」
2. 模板部署教學不存在：`create-service` 連結到 `/docs/template/deploy-template`，但該頁面是 404。模板是最低門檻的入口，卻沒有完整教學
3. 部署流程散落各處：沒有一篇完整的 reference 讓 AI 能快速索引
4. 部署區沒有按產品架構分層：Zeabur 本身是「伺服器 → 專案 → 服務」三層，但文檔把專案和服務層混在同一個 `deploy/`，管理功能又放在另一個 `manage/`
5. AI 對話窗口沒有引導文：用戶不知道它能做什麼、什麼時候該用

## 分類原則

不按功能分類，按**查閱動機**分類：

| 動機 | 讀者 | 文件類型 |
|------|------|---------|
| 「我想馬上試試看」 | 新用戶 | Human-first quickstart |
| 「我想知道能做什麼」 | 評估中的用戶 | Human-first 導覽 |
| 「我要部署 X 框架」 | 開發者 / AI agent | AI-first reference |
| 「我的部署壞了」 | 開發者 / AI agent | AI-first troubleshooting |
| 「我要管帳單」 | 付費用戶 | Human-first 設定 |
